## 评分依据

### F1：极简输入 & 一键生成行程 — ICE 90

| 维度           | 评分 | 一句话依据                                                                                                                                                         |
| -------------- | ---- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Impact**     | 10   | 产品的核心价值——没有行程生成，就没有产品。直接解决"信息过载"这个首要痛点，将 3-5 小时规划时间压缩至 30 秒。                                                        |
| **Confidence** | 9    | 大模型 API 已非常成熟（GPT-4、Claude 均支持结构化输出），竞品（ChatGPT、飞猪问一问、Roam Around）已验证此路径可行，且脑暴调研显示 91% 旅行者已在使用 AI 辅助规划。 |
| **Ease**       | 10   | 实现路径清晰：一个输入表单 + 大模型 API 调用 + JSON 解析渲染。技术上无难点，3-4 天可完成 MVP。                                                                     |

### F3：对话式微调 — ICE 70（↓20，Confidence 下调）

| 维度           | 评分        | 一句话依据                                                                                                                                                                                                                                                               |
| -------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Impact**     | 10          | 直接解决"调整不灵活"这个第二核心痛点——用户用自然语言说"第二天下午换个轻松点的"就能调整行程，这是区别于所有竞品的核心体验（Wanderlog 无自然语言编辑，携程问道调整自由度有限）。                                                                                           |
| **Confidence** | **7**（↓2） | **修订**：多轮对话是大模型的强项，但真实用户体验中存在额外风险：(1) 旅途中打字不便，(2) 用户与 AI 对"轻松"等主观描述的理解可能不同，(3) 长对话上下文丢失，(4) 误删操作恢复成本高。技术上可行不代表体验上可靠。建议优先验证删除和全局调整两类高风险意图的 prompt 准确率。 |
| **Ease**       | 10          | 基于大模型的多轮对话能力，核心工作是 prompt 模板设计和上下文管理。前端为半屏对话面板，复杂度低。                                                                                                                                                                         |

### F2：行程卡片系统 — ICE 63（↑14，Impact 上调）

| 维度           | 评分        | 一句话依据                                                                                                                                                                                                                                                                               |
| -------------- | ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Impact**     | **9**（↑2） | **修订**：行程卡片系统不是"展示"——它是用户的**判断工具**。卡片的信息架构直接决定了用户能否快速判断"行程是否合理"。如果卡片做不好——看不到交通时间、总步行距离——用户就不得不切到高德地图自己验证，这正是 PRD 中要解决的"信息过载"问题的延续。F2 还承载了替换/删除/标记"必去"三个决策行为。 |
| **Confidence** | 7           | 卡片式 UI 是成熟的交互模式，但卡片内容的丰富度（照片、评分、门票价格）依赖 POI 数据源质量。V1.0 可先不做实景照片，降低对 POI 数据的依赖。                                                                                                                                                |
| **Ease**       | 10          | 卡片列表是前端基础组件，技术复杂度低。主要工作在 UI 设计和信息层级梳理。                                                                                                                                                                                                                 |

### F6：行程管理（保存/导出/历史） — ICE 33.6

| 维度           | 评分 | 一句话依据                                                                                                                               |
| -------------- | ---- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| **Impact**     | 6    | 让旅途中可以随时查看已规划行程，支持长图导出和微信分享给同伴。没有此功能产品仍可使用（行程在会话中可见），但旅途体验和分享能力会打折扣。 |
| **Confidence** | 8    | 自动保存、列表管理、图片导出、微信分享都是微信小程序的标准能力，实现确定性高。唯一不确定点是长图渲染在不同机型上的兼容性。               |
| **Ease**       | 7    | 涉及本地存储 + 云端同步（需设计存储方案）、图片渲染导出、微信分享 API 接入，比前三个功能复杂度略高。约 2-3 天。                          |

### 🆕 F7：POI 数据实时富化 — ICE 24

| 维度           | 评分  | 一句话依据                                                                                                                                                                                                                                                    |
| -------------- | ----- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Impact**     | **8** | 直接解决"AI 会不会瞎编景点"这个信任锚点。旅游决策场景下，可信度是留存的前提——用户一旦发现 AI 推荐的景点不存在或已关闭，就再也不会用了。高德 POI 校验 + 真实照片/评分/价格，让每张卡片从"AI 说"变成"数据证明"。                                                |
| **Confidence** | **6** | 存在三个不确定性：(1) 中文 POI 名称模糊匹配比预期棘手（AI 生成的"西湖"和高德的"西湖风景名胜区"不一定直接命中）；(2) 照片 URL 在微信小程序 image 组件中的加载兼容性待验证；(3) prompt 名称标准化效果需实测。建议先做最小验证版本（3 个城市 × 10 个热门 POI）。 |
| **Ease**       | **5** | 涉及云函数改造（并发调用 + Provider 抽象）、缓存层设计（POI ID 索引）、Prompt 工程（名称标准化 + few-shot）、前端渐进渲染改造（loading/success/error 三态）。5.5 人天，不是"一个下午能搞定"的量。                                                             |

### F4：拖拽排序编辑 → **V1.0 移除**

| 维度           | 评分        | 一句话依据                                                                                                                         |
| -------------- | ----------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| **Impact**     | **5**（↓1） | **修订**：V1.0 仅做静态标注（气泡 + 距离），不做路线连线、拖拽联动。虽然 Wanderlog 的体验很好，但微信小程序原生 map 组件难以复现。 |
| **Confidence** | 6           | 微信小程序地图组件功能有限，但静态标注技术风险可控。                                                                               |
| **Ease**       | **9**（↑2） | **修订**：从 4 人天压缩到 1.5 人天（静态标注版），仅标注点位和距离。Ease 大幅提升。                                                |

### F4：拖拽排序编辑 → **V1.0 移除**

> **决策记录（2026-07-02）**：从 V1.0 移除。理由：(1) ICE 评分最低（原 10 分）；(2) 小程序中实现流畅拖拽的技术风险高（Ease=4）；(3) F3 对话微调已覆盖"调整顺序"场景；(4) 节省 5 人天可投入 DTO 校验层、静态地图、首次用户引导等更高价值功能。

## 四象限分析（影响 x 容易度）

| 象限           | 影响      | 容易度    | 策略                         | 需求列表                              |
| -------------- | --------- | --------- | ---------------------------- | ------------------------------------- |
| **Quick Wins** | 高（>=7） | 高（>=7） | 优先做，性价比最高           | F1 一键生成、F3 对话微调、F2 卡片系统 |
| **Strategic**  | 高（>=7） | 低（<7）  | 规划做，值得投入但需仔细设计 | —（V1.0 无此象限需求）                |
| **Fill-ins**   | 低（<7）  | 高（>=7） | 有空就做，成本不高且有价值   | F6 行程管理、F5 地图（静态标注版）    |
| **Cautious**   | 低（<7）  | 低（<7）  | 谨慎做，需评估是否值得投入   | F4 拖拽排序 → **已推迟 V2.0**         |

**洞察**：V1.0 的 6 个功能中，有 3 个是 Quick Wins（F1/F2/F3），这意味着产品可以用很低的成本快速搭建出可用的核心体验。F5 和 F4 属于 Cautious 象限，需要认真考虑是否真的需要在 V1.0 中实现。

## 决策记录

### 核心取舍

**选了 F3（对话微调）作为核心编辑方式，F4（拖拽排序）已从 V1.0 移除**：对话微调覆盖替换/新增/删除/全局调整/调整顺序等多种意图，且在自然语言表达上更灵活。拖拽排序在小程序环境中的实现风险高（5 人天、手势冲突多），且功能上与对话微调重叠。经专家团队综合评估，决定 V1.0 专注对话微调，拖拽排序推迟到 V2.0。

**F5（地图视图）降级为静态标注版**：做 1.5 人天的轻量版（仅标注气泡 + 距离），不做路线连线、拖拽联动和日期聚焦动画。在微信小程序原生 map 组件限制下，这是最务实的方案。完整的路线可视化留给 V2.0。

### 争议需求

**F4 拖拽排序 —— 已决策移除**：在四份文档中优先级矛盾（PRD 标 P0，ICE 打分最低，用户故事放 Phase 3）。经专家团队评估，确认从 V1.0 移除。节省的 5 人天分配到 DTO 校验层（2天）、首次用户引导（1天）和缓冲（2天）。

**F5 地图视图 —— 已决策降级**：V1.0 仅做静态标注版（1.5 人天），从 Cautious 象限移至 Fill-ins 象限。完整的路线可视化和拖拽联动留到 V2.0。

### 信心标注（C < 7 的需求）

| 需求        | Confidence    | 建议验证方式                                                      |
| ----------- | ------------- | ----------------------------------------------------------------- |
| F5 地图视图 | 6             | 做静态标注版原型，测试在低端机上的渲染性能                        |
| F3 对话微调 | **7**（修订） | 优先验证删除和全局调整两类高风险意图的 prompt 准确率（需 >= 85%） |

### 下期候选（V2.0 中可能提前晋升的功能）

如果 V1.0 Phase 1 验证顺利，**旅途实时助手-即时问答**最可能提前进入开发——它复用 F3 的对话能力，只是把对话场景从"规划阶段"延伸到"旅途阶段"，边际成本低但体验提升大。

## 排序假设前提

1. **开发者能力**：假设 1 名熟悉微信小程序和大模型 API 的全栈开发者
2. **大模型选择**：假设使用 GPT-4 或同等能力模型，支持结构化 JSON 输出
3. **个人使用场景**：假设不需要考虑多用户并发、运营后台、商业化等功能
4. **数据源**：假设高德或腾讯地图 POI API 可免费使用且满足需求
5. **微信环境**：假设微信小程序审核政策不会限制 AI 生成内容类应用

> 如以上假设发生变化，评分可能需要重新校准。
