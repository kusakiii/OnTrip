# ADR-005：缓存策略 — POI ID 索引 + 云数据库临时集合

**状态**：已接受
**日期**：2026-07-03

## 背景

POI 富化需要调用高德 API。同一城市的景点（如"西湖断桥"）可能被多次查询（不同用户、不同风格的行程）。需要缓存减少 API 调用和响应延迟。

## 决策

- **缓存 Key**：高德 POI ID（如 `B000A8U5Y8`），**不是** AI 生成的名称
- **存储**：云数据库临时集合 `poi_cache`
- **TTL**：7 天
- **写入策略**：异步写，失败不影响主流程

## 理由

AI 对同一景点可能输出多种名称（"西湖" / "西湖断桥" / "断桥残雪"），以名称为 Key 会导致多次 cache miss。先查高德拿到官方 POI ID，再用 ID 做 Key，才能真正命中缓存。

## 后果

- ✅ 热门目的地缓存命中率 60%+，每次命中节省 ~400ms
- ❌ 首次查询仍需调高德 API，无法完全避免
- ❌ 缓存写入失败仅记录日志，下次仍会查 API（降级而非阻塞）
