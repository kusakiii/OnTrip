# 产品脑暴记录

**主题**：AI 旅程规划应用——AI 生成行程 + 用户灵活调整
**HMW**：我们如何能让休闲度假旅行者在 AI 快速生成一份可靠行程的基础上，轻松灵活地根据个人偏好和旅途变化进行调整？
**日期**：2026-07-02
**产出创意数**：24 个（筛选后保留 18 个）
**使用框架**：SCAMPER / 5 Why / 竞品启发 / 约束创新 / 反向脑暴

---

## 问题聚焦

**用户画像**：个人用户，休闲度假风格，希望省心但保留灵活度
**核心痛点**：

1. 信息过载导致决策困难——攻略太多、推荐太泛，不知道该信谁
2. 行程中缺乏灵活调整能力——计划赶不上变化，AI 帮不上忙
   **AI 期望**：先生成完整行程，用户在此基础上判断和调整
   **产品形态**：待定（先发散功能创意，形态后置）

## Top 3 推荐创意

### 创意1：AI 一键生成 + 对话微调 + 卡片拖拽（组合方案）

- **描述**：用户输入"目的地+天数+风格标签"三个要素，AI 在 30 秒内生成一份结构化的完整行程，以可拖拽的卡片形式呈现。每张卡片代表一个行程节点（景点、餐厅、交通段），用户可以通过三种方式调整：(1) 直接拖拽排序/删除/替换；(2) 用自然语言对话（"第二天下午换个轻松点的"）；(3) 点击卡片查看详细信息后手动微调。三种编辑方式互为补充，覆盖从"粗略调整"到"精细打磨"的全场景。
- **解决的痛点**：信息过载（AI 替你做 80% 的决策）+ 调整不灵活（三种编辑方式任选）
- **预期效果**：将旅行规划时间从平均 3-5 小时缩短至 15-30 分钟；用户对生成行程的满意度预期达 70%+（首次生成），经调整后达 90%+
- **实现路径**：
  - 后端：接入大模型 API（如 GPT-4 / Claude），设计行程生成的 prompt 模板和结构化输出 Schema（JSON）
  - 前端：卡片式行程 UI，支持拖拽排序（React DnD / SortableJS）+ 对话输入框
  - 数据层：景点 POI 数据库 + 地图路线 API（验证空间合理性）
  - MVP 阶段可先做纯对话式，再加卡片拖拽
- **验证方式**：先用 ChatGPT API + 简单前端做原型，自己规划 3 次真实旅行测试体验，迭代 prompt 和交互
- **风险**：大模型可能产生幻觉（推荐已关闭的景点）→ 初期人工审核 + 逐步接入实时数据源校验

### 创意2：弹性行程块 + 风格一键切换

- **描述**：打破传统精确到小时的排程方式，采用"上午块 / 下午块 / 晚间块"的弹性时间块组织行程。每个时间块内提供 2-3 个可选项（如"下午：A. 博物馆 B. 海边散步 C. 咖啡馆发呆"），用户可按心情选择。同时支持"一键风格切换"——对同一份行程，点击"文艺版 / 美食版 / 暴走版 / 佛系版"按钮，AI 自动重新编排对应风格的行程。
- **解决的痛点**：行程过于刚性不好调整（弹性块天然支持灵活选择）+ 信息过载（风格切换帮用户快速看到不同方案）
- **预期效果**：用户在旅途中的行程调整频率预计降低 50%（因为每块本身就有备选项）；风格切换功能预计提升首次生成满意度 15-20%
- **实现路径**：
  - 行程 Schema 从"时间线"改为"时间块 + 选项列表"
  - 风格标签系统：定义 4-5 种旅行风格，每种风格对应不同的 prompt 策略和 POI 筛选规则
  - 前端：时间块卡片 UI，每块内 Tab 切换不同选项 + 顶部风格切换按钮
- **验证方式**：为同一个目的地生成"弹性版"和"精确版"行程各 5 份，分别体验后对比满意度和实际调整次数
- **风险**：选项太多可能反而加重决策负担 → 每块严格控制为 2-3 个选项，并用 AI 标注"推荐"项

### 创意3：旅途实时助手（轻量版 MVP）

- **描述**：在旅途执行阶段，AI 助手持续运行，重点解决三个高频场景：(1) 天气突变——检测到下雨/极端高温时，自动将户外行程替换为室内备选并推送通知；(2) 路线优化——基于用户当前位置和剩余行程，建议更优的游览顺序；(3) 即时问答——"附近有什么好吃的"、"这个景点排队多久"等实时问题。MVP 阶段先做天气触发 + 即时问答两个场景。
- **解决的痛点**：行程中缺乏灵活调整（AI 主动感知变化并推送方案）
- **预期效果**：旅途中用户遇到问题时，不再需要自己搜攻略，预计减少 60% 的"计划外焦虑"
- **实现路径**：
  - 接入天气 API（如 OpenWeatherMap），设置触发阈值（降雨概率 > 60%）
  - 接入地图定位 API，获取用户当前位置
  - 大模型 API 处理即时问答，结合目的地 POI 数据给出推荐
  - 通知推送：App 内通知 或 微信消息推送
  - MVP 可先做微信小程序（开发成本低，推送方便）
- **验证方式**：下一次旅行中实际使用，记录"AI 推送有用 / 无用 / 缺失"的次数和场景
- **风险**：实时数据接入成本和数据准确性 → MVP 阶段只接天气 API，其余用大模型 + 搜索引擎兜底

## 被放弃的创意（存档参考）

- **实时价格追踪**：放弃原因——价格 API 接入成本高、数据源分散（各平台不开放），作为个人工具投入产出比太低，标记为 Money Pit
- **多人协作投票**：放弃原因——用户明确是"自己用"，多人场景不在 MVP 范围内

### 5 Why 根因追问

**痛点1：信息过载难决策**

- Why 1：选项太多，没有有效的筛选标准
  - Why 2：工具不了解用户偏好，只能罗列所有可能性
    - Why 3：缺少个性化的用户画像和历史数据积累
      - Why 4：每次旅行都从零开始，系统不学习
        - Why 5（根因）：**工具缺乏"认识你"的能力**
        - 方案 → 旅行人格画像（创意9）+ 风格标签系统（创意16）
    - 方案 → 对话式偏好引导 + 排除式规划（创意8）
  - 方案 → AI 主动推荐排序，附带推荐理由（创意7）
- 方案 → 极简输入一键生成，AI 替你做 80% 决策（创意1）

**痛点2：行程中缺乏灵活调整**

- Why 1：行程是静态的，旅途变化无法被感知
  - Why 2：系统不连接实时数据源（天气/交通/客流）
    - Why 3：规划和执行是割裂的两个阶段
      - Why 4（根因）：**产品只做"规划工具"，不做"随行服务"**
      - 方案 → 旅途实时助手（创意6）
    - 方案 → 弹性行程块，每块预留备选项（创意11）
  - 方案 → 基于位置的即时问答（创意6-场景3）
- 方案 → 对话式微调 + 卡片拖拽，调整门槛极低（创意2、3）

| 恶化方案                                 | 逆转后的改进方案                                    |
