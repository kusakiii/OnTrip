# 途灵（TripFlow）技术选型方案

**日期**：2026-07-02
**产品定位**：个人使用的 AI 旅程规划微信小程序

---

## 1. 技术需求分析

基于 PRD 中的 18 个 User Story，核心技术需求可归纳为以下五层：

| 层次     | 需求                                           | 关键约束                        |
| -------- | ---------------------------------------------- | ------------------------------- |
| 前端框架 | 微信小程序，支持复杂交互（拖拽、地图、对话）   | 性能、手势交互流畅度            |
| AI 引擎  | 行程生成、对话微调、意图识别、结构化 JSON 输出 | 中文能力、JSON 输出稳定性、成本 |
| 后端服务 | API 代理（密钥隔离）、行程 CRUD、用户认证      | 轻量、低运维、个人项目成本可控  |
| 数据存储 | 行程 JSON 文档存储、本地离线缓存               | 文档型数据、离线可用            |
| 地图服务 | POI 搜索、路线绘制、定位、地理编码             | 微信小程序原生兼容              |

## 2. 前端框架

### 推荐：原生微信小程序开发

| 对比项            | 原生微信小程序        | Taro 4.x               | uni-app x                |
| ----------------- | --------------------- | ---------------------- | ------------------------ |
| 地图+拖拽交互性能 | 最优（原生组件）      | 有损耗，需额外适配     | 较好，但有兼容坑         |
| 开发效率          | WXML 模板语法，上手快 | React 生态，需配置编译 | Vue 生态，HBuilderX 绑定 |
| 地图 SDK 兼容性   | 腾讯地图 SDK 天然适配 | 需条件编译，踩坑多     | 较好，有插件但质量不一   |
| 跨端能力          | 仅微信                | 多端支持               | 多端支持                 |
| 包体积            | 最小                  | 运行时库增加体积       | UTS 编译较优             |

**选择理由**：

途灵是个人工具，仅需微信小程序单端发布。核心交互（US-11/12 拖拽排序、US-13/14 地图路线渲染）对性能敏感，原生组件（`movable-view`、`map`）在这些场景下表现最好，不存在框架抽象层的兼容性风险。同时省去了跨端框架的编译配置和调试成本，对个人项目来说更直接。

**UI 组件库**：采用 **Vant Weapp**（有赞出品），提供成熟的弹窗、Toast、ActionSheet、Tab 等基础组件，与原生小程序兼容性好。

**状态管理**：使用 **MobX Miniprogram Bindings**（微信官方适配），轻量够用，比 Redux 方案简单得多。

## 3. AI 大模型

### 推荐：DeepSeek V4 Flash（主） + Qwen3-Plus（备）

| 对比项                    | DeepSeek V4 Flash | Qwen3-Plus     | ERNIE 4.5 Turbo | Kimi K2.6      |
| ------------------------- | ----------------- | -------------- | --------------- | -------------- |
| 输入价格（元/百万 token） | 0.02（非高峰）    | 1.6            | 0.8             | 4.3            |
| 输出价格（元/百万 token） | 2（非高峰）       | 6.4            | 3.2             | 21             |
| 中文旅行知识              | 优秀              | 优秀           | 优秀            | 良好           |
| 结构化 JSON 输出          | 稳定              | 稳定           | 稳定            | 稳定           |
| OpenAI 兼容格式           | 完全兼容          | 兼容           | 部分兼容        | 完全兼容       |
| 响应速度                  | 60-80 tokens/s    | 50-70 tokens/s | ~50 tokens/s    | 30-50 tokens/s |
| 免费额度                  | 较少              | 7000 万 tokens | 少量体验额度    | ~15 元额度     |

**选择理由**：

**DeepSeek V4 Flash** 作为主力模型：非高峰期输入价格低至 0.02 元/百万 token，生成一份 3 天行程（约 3000 token 输入 + 2000 token 输出）成本不到 0.01 元。个人使用场景下月成本几乎可以忽略。OpenAI 兼容格式意味着切换模型只需改 `base_url` 和 `api_key`。

**Qwen3-Plus** 作为备选：通义千问免费额度高达 7000 万 tokens，足够开发调试和早期使用。且阿里的旅行生态（高德、飞猪）使其在旅行知识上略有优势。当 DeepSeek 服务波动时可无缝切换。

**统一接口层设计**：后端封装一个 `LLMService` 类，内部使用 OpenAI SDK 统一封装，通过配置切换 `base_url`，支持运行时 fallback（主模型超时自动切备用）。

```
用户请求 → LLMService → DeepSeek V4 Flash（首选）
                        ↓ 超时/失败
                      Qwen3-Plus（备选）
```

## 4. 后端服务

### 推荐：微信云开发（CloudBase）

| 对比项       | 微信云开发                | 腾讯云轻量服务器      | Cloudflare Workers     |
| ------------ | ------------------------- | --------------------- | ---------------------- |
| 与小程序集成 | 原生 wx.cloud API，零配置 | 需自建 API + 域名备案 | 需自建 API，国内访问慢 |
| 运维成本     | 零运维                    | 需自行维护服务器      | 低运维                 |
| 免费额度     | 足够个人使用              | 最低 79 元/年         | 免费但国内受限         |
| 冷启动       | 100-500ms（可接受）       | 无冷启动              | <5ms                   |
| 域名/备案    | 不需要                    | 需要 ICP 备案         | 不需要（但国内慢）     |
| 数据库       | 内置 NoSQL                | 自建                  | D1（SQLite）           |

**选择理由**：

个人工具项目的黄金标准——零运维、零备案、免费额度充足。`wx.cloud.callFunction` 从小程序直接调用云函数，不需要额外配置域名和 HTTPS。云函数负责代理 AI 请求（隐藏 API Key）和行程数据 CRUD，内置的 NoSQL 数据库天然适合存储行程 JSON 文档。

**云函数设计**（3 个函数即可）：

| 云函数         | 职责                                                                 | 触发场景             |
| -------------- | -------------------------------------------------------------------- | -------------------- |
| `ai-chat`      | 代理 AI 请求，注入系统 prompt，**含 DTO 校验层**，调用 DeepSeek/Qwen | 生成行程、对话微调   |
| `trip-manager` | 行程 CRUD（创建/读取/更新/删除）                                     | 保存、加载、删除行程 |
| `poi-search`   | 代理地图 POI 搜索请求                                                | 搜索景点、餐厅等     |

**DTO 校验层设计**（ai-chat 云函数内部）：

```
AI 原始响应 → JSON 解析 → Schema 校验 → 自动修复 → 返回前端
                              ↓ 失败
                         重试 1 次 → 仍失败? → 降级模板
```

校验规则（基于 PRD 附录 A 的 Schema）：

1. `itinerary[].cards[]` 不为空（每天至少 1 个卡片）
2. `location.lat/lng` 为合法数值（范围内）
3. `timeRange` 格式符合 `HH:MM-HH:MM`
4. 缺失字段赋默认值（不抛异常）
5. 多余字段保留（向前兼容），未知字段记录警告日志

## 5. 数据存储

### 推荐：微信云数据库 + 本地 Storage 双层架构

```
┌─────────────┐     ┌──────────────┐
│  小程序端    │     │   云开发后端   │
│             │     │              │
│ wx.setStorage│────│  云数据库     │
│ (本地缓存)   │     │ (NoSQL)      │
│             │     │              │
│ 离线时读取    │←───│ trip-manager │
│ 本地缓存     │     │ 云函数        │
└─────────────┘     └──────────────┘
```

**数据结构设计**（行程文档）：

```json
{
  "_id": "trip_xxx",
  "_openid": "用户 openid",
  "destination": "杭州",
  "days": 3,
  "styleTags": ["美食", "文艺"],
  "createdAt": "2026-07-02T10:00:00+08:00",
  "updatedAt": "2026-07-02T11:30:00+08:00",
  "itinerary": [
    {
      "dayIndex": 1,
      "date": "2026-07-10",
      "cards": [
        {
          "cardId": "card_001",
          "type": "spot",
          "name": "西湖断桥",
          "timeRange": "09:00-11:00",
          "duration": 120,
          "description": "西湖十景之一...",
          "location": { "lat": 30.259, "lng": 120.149 },
          "openHours": "全天",
          "ticketPrice": "免费",
          "rating": 4.6,
          "transportFromPrevious": null,
          "isStarred": false,
          "priority": "optional"
        }
      ]
    }
  ]
}
```

**离线策略**：每次保存行程时，同步写入 `wx.setStorage`。用户打开小程序时先加载本地缓存，再异步拉取云端最新版本。

> ⚠️ **V1.0 离线能力边界**：离线模式下可查看**行程文字概要**（景点名称、时间安排），但地图、实景照片、AI 对话功能不可用。编辑操作暂存本地，联网后采用"最后写入胜出"（Last-Write-Wins）策略同步。V2.0 考虑增加"行前预加载"（Wi-Fi 下自动下载地图瓦片和 POI 数据）。

## 6. 地图服务

### 推荐：腾讯地图 SDK（微信小程序原生）

微信小程序内置 `map` 组件底层就是腾讯地图，无需额外集成。POI 搜索和地理编码通过腾讯地图 WebService API 调用（在云函数中代理，避免暴露 Key）。

**使用的核心能力**：

- `map` 组件：路线渲染、标注气泡、缩放控制
- `movable-view` 组件：拖拽排序的手势交互
- 腾讯地图 WebService API（云函数代理）：POI 搜索、地理编码、路线规划

## 9. 技术风险与应对

| 风险                                | 概率 | 影响 | 应对策略                                                                                                |
| ----------------------------------- | ---- | ---- | ------------------------------------------------------------------------------------------------------- |
| AI 生成内容含幻觉（已关闭的景点等） | 高   | 中   | 云函数中对返回结果做 POI 校验（调腾讯地图 API 确认存在性）；卡片详情标注"信息仅供参考"                  |
| 大模型 API 响应超时（>30s）         | 中   | 低   | 设置超时 + 自动重试 + 双模型 fallback；生成过程展示进度文案降低等待焦虑                                 |
| 大模型 JSON 输出格式异常            | 中   | 高   | 云函数中增加 DTO 校验层（Schema 校验 + 自动修复 + 降级模板）；前端只接收校验通过的数据                  |
| 高德照片 URL 在微信小程序中加载失败 | 高   | 中   | 🆕 上线前真机验证 10+ 个图片 URL；前端 binderror 降级为类型占位图；必要时增加云函数图片代理端点         |
| POI 名称匹配准确率不足              | 中   | 中   | 🆕 Prompt 增加名称标准化约束 + few-shot 示例；后端城市约束强制 + 编辑距离相对阈值；缓存命中减少重复查询 |
| 微信云开发数据库配额不足            | 低   | 低   | 个人使用场景数据量极小；必要时可清理历史行程或升级为付费版                                              |
| 小程序审核不通过（AI 内容类）       | 低   | 中   | 提前了解旅游类目资质要求；准备人工审核备用方案；个人主体可能受限，需确认                                |
| 多模型 fallback 时 prompt 不兼容    | 中   | 中   | 对 DeepSeek 和 Qwen 分别做 50 个场景的 A/B 对比测试；为两个模型维护差异化的 prompt 模板                 |

> **已移除风险**：小程序拖拽交互不流畅 → V1.0 已移除拖拽功能，不适用。

## 10. 开发路线图（修订版：8 周）

> **修订说明（2026-07-02）**：F4 拖拽排序已移除，地图降级为静态标注版，增加 PoC 周和缓冲周。

| 阶段                         | 时间          | 目标                        | 关键里程碑                                         |
| ---------------------------- | ------------- | --------------------------- | -------------------------------------------------- |
| **PoC 验证**                 | 第 1 周       | 验证核心技术可行性          | 原生小程序首页 PoC + Schema 定稿 + Prompt Playbook |
| **Phase 1**                  | 第 2-4 周     | 核心闭环可用                | 生成→卡片→对话→保存（不含拖拽）                    |
| **真实验证**                 | 第 4 周末     | 用 MVP 规划一次真实短途旅行 | 收集体验反馈，确认核心假设                         |
| **Phase 2**                  | 第 5-6 周     | 体验补全                    | 静态地图 + 导出分享 + 历史管理 + 首次引导          |
| **缓冲 & 上线**              | 第 7 周       | 审核 + Bug 修复             | 微信审核提交、prompt 调优、Bug 清零                |
| **F7 POI 富化（P1 后半段）** | 第 6-7 周并行 | 行程数据可信度增强          | 高德 API 集成 + Prompt 标准化 + 前端渐进渲染       |
| **V2.0 规划**                | 第 8 周+      | 旅途实时助手                | 基于 V1.0 数据决定优先级                           |
