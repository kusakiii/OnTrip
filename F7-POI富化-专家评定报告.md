# F7 POI 数据实时富化 — 专家团队评定报告

**审查日期**：2026-07-03
**审查团队**：产品经理、后端架构师、微信小程序开发者、UX 架构师、软件架构师、Prompt 工程师、代码审查专家

---

## 一、总体评定

**7 位专家一致同意这个功能方向**——用高德 POI API 校验和补全 AI 生成的行程数据，是解决"AI 推荐是否可信"这个核心信任问题的务实方案。

**但是，原始设计有 5 处需要关键修改，1 处严重代码风险，1 处需要上线前验证。**

---

## 二、致命问题（必须修复）

### 🔴 Critical 1：catch 吞噬一切 + 已富化数据被错误标回

**发现者**：代码审查专家

```javascript
// ❌ 原始伪代码的问题
try {
  card.rating = best.rating || card.rating;     // 已写入
  card.photos = best.photos || [];               // 已写入
  await setCache(cacheKey, {...});                // ← 缓存写失败
} catch (err) {
  card._poiVerified = false;  // ⚠️ 数据已富化，却被标为"未验证"
  card._dataSource = "ai";    // ⚠️ 明明是 amap 数据，却标为 ai
}
```

**修复**：缓存写入独立 try-catch，不影响已富化的数据；区分程序员错误（throw）和网络错误（降级）。

### 🔴 Critical 2：缺少输入校验

```javascript
async function enrichWithPOI(trip, destination) {
  for (const day of trip.itinerary) { ... } // trip=null → TypeError 被 catch 吞掉
```

**修复**：函数入口加 `if (!trip?.itinerary?.length) throw new TypeError(...)`，快速失败。

### 🔴 Critical 3：照片 URL 在微信小程序中可能全部无法加载

**发现者**：后端架构师、微信小程序开发者

高德图片 URL（`aos-cdn.amap.com`）可能存在：Referer 防盗链拦截、ToS 合规风险、URL 时效性。

**修复**：**上线前必须做**——在微信小程序真机上测试 10 个不同高德图片 URL 的加载成功率。如果不通过，增加云函数图片代理端点。

## 三、设计修改（高优先级）

### 🟠 修改 1：串行 → 并行化调用（所有专家共识）

|          | 原始设计                | 修改后                            |
| -------- | ----------------------- | --------------------------------- |
| 调用方式 | 15 张卡片逐个串行       | `Promise.all` + `p-limit(5)` 并发 |
| 耗时     | 2-30s（取决于尾部延迟） | ~1-2s                             |
| 超时风险 | 尾部延迟场景逼近 60s    | 安全                              |

### 🟠 修改 2：缓存 Key 从 AI 名称 → 高德 POI ID

**发现者**：产品经理

原始 Key `{城市}_{POI名称}_{类型}` 的问题：AI 对同一景点可能输出不同表述（"拙政园" / "拙政园（苏州园林代表）" / "苏州拙政园"）→ 三个 cache miss。

**修复**：先用搜索名查高德 → 拿到官方 POI ID → 用 POI ID 做缓存 Key。

### 🟠 修改 3：匹配策略增强（Prompt 侧 + 后端侧）

**发现者**：Prompt 工程师（最关键发现）

| 修改                        | 匹配率提升 | 成本                      |
| --------------------------- | ---------- | ------------------------- |
| Prompt 增加"名称标准化"约束 | +30-40%    | ~500-800 tokens           |
| AI 输出高德分类码           | +10-15%    | ~120-150 tokens           |
| Few-shot 示例（3-5 个）     | +10-17%    | ~150-250 tokens（一次性） |
| 双模型差异化 prompt         | +10-15%    | ~200-400 tokens（选择性） |

**综合效果**：匹配率从 ~35% 提升到 **85%+**。成本增量 < 0.001 元/次。

### 🟠 修改 4：前端渐进渲染 + 统一降级态

**发现者**：产品经理、UX 架构师

- 先渲染 AI 骨架卡片 → 富化完成后原地"点亮"（照片浮现、评分弹出）→ 感知延迟大幅缩短
- `_poiVerified: false` 的卡片统一显示"信息收集中"占位态，不暴露空白字段
- 不直接标注"未验证"等负面工程标签

### 🟠 修改 5：Provider 接口抽象

**发现者**：软件架构师、后端架构师

```javascript
// 标准化 Provider 接口
const amapProvider = createAmapProvider(apiKey);     // 高德
const tencentProvider = createTencentProvider(apiKey); // 腾讯（未来）

// 富化层只消费接口，不感知具体厂商
async function enrichPOI(card, provider, options) { ... }
```

## 四、优先级和工时估算

**产品经理 ICE 评估**：

| 维度       | 评分 | 理由                                            |
| ---------- | ---- | ----------------------------------------------- |
| Impact     | 8/10 | 直接解决"AI 会不会瞎编景点"的信任锚点           |
| Confidence | 6/10 | 匹配准确性存不确定性，需 prompt 优化后验证      |
| Ease       | 5/10 | 涉及云函数改造、缓存、前端渐进渲染、prompt 工程 |

**建议**：P1 后半段（Phase 2 第 5-6 周），先做最小验证版本（3 个城市 × 10 个热门 POI）。

**工时估算**：

| 子任务                                                | 工时         |
| ----------------------------------------------------- | ------------ |
| POI 富化层核心逻辑（并行调用 + 降级 + Provider 接口） | 2 人天       |
| Prompt 工程（名称标准化 + few-shot + 双模型适配）     | 1.5 人天     |
| 缓存层（POI ID 索引 + 7 天 TTL）                      | 0.5 人天     |
| 前端渐进渲染 + 降级态                                 | 1 人天       |
| 照片 URL 真机验证 + 兜底方案                          | 0.5 人天     |
| **合计**                                              | **5.5 人天** |

## 五、修改后的数据流

```
用户点击「生成行程」
        │
        ▼
   ai-chat 云函数
        │
   DeepSeek 生成行程骨架
   （Prompt 已含名称标准化约束 + 分类码 + few-shot）
        │
   DTO 校验层
        │
   🆕 POI 富化层
   ├── Promise.all(p-limit=5) 并行查询高德
   ├── 每张卡片独立 try-catch（缓存→API→匹配→富化）
   ├── 缓存 Key = 高德 POI ID（非 AI 名称）
   ├── 失败降级：_poiStatus = "verified" | "not_found" | "error"
   └── Provider 接口抽象（方便换源）
        │
   返回富化后的行程 JSON
        │
   前端渐进渲染
   ├── 先展示骨架卡片（灰色占位）
   ├── POI 数据就绪后「点亮」（照片浮现、评分弹出）
   └── 未匹配卡片统一"信息收集中"占位态
```
