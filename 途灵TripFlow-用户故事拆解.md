# 用户故事拆解：途灵（TripFlow）V1.0

**日期**：2026-07-02
**拆解范围**：V1.0 全部功能（Phase 1-3）
**粒度**：Sprint 级（每个 Story 1-3 天）
**估算体系**：Fibonacci（1/2/3/5/8/13）
**总 Story 数**：20 个
**总 Story Points**：65

---

## 需求全貌

**涉及角色**：旅行者（唯一角色，个人工具）

**核心流程**：输入目的地/天数/风格 → AI 生成行程 → 卡片浏览 → 对话/拖拽调整 → 地图查看 → 保存/导出/分享 → 旅途中查看

**数据实体**：行程（Trip）→ 天（Day）→ 卡片（Card）→ 交通段（Transport）

**拆分模式选择**：

| 功能                   | 拆分模式                      | 理由                                          |
| ---------------------- | ----------------------------- | --------------------------------------------- |
| F1 极简输入 & 一键生成 | 按工作流步骤 + 按复杂度递进   | 流程明确（输入→生成→反馈），需区分 MVP 和增强 |
| F2 行程卡片系统        | 按工作流步骤 + 按 CRUD        | 浏览→查看详情→操作（替换/删除/标记）          |
| F3 对话式微调          | 按业务规则变体 + 按复杂度递进 | 7 种对话意图按复杂度分组                      |
| F4 拖拽排序            | 按业务规则变体                | 同日排序 vs 跨天拖拽，规则不同                |
| F5 地图视图            | 按工作流步骤 + 按复杂度递进   | 展示→聚焦→交互                                |
| F6 行程管理            | 按 CRUD + 按数据操作          | 保存/导出/查看/复制/删除                      |

---

## Epic 1：极简输入 & 一键生成行程（F1）

### US-01：输入信息并生成行程（MVP）

**角色**：作为旅行者
**需求**：我希望输入目的地、天数和旅行风格标签后，AI 自动生成一份完整的行程
**价值**：省去翻阅大量攻略的时间，30 秒内获得一份可靠的行程草案
**优先级**：P0
**Story Points**：5

**Acceptance Criteria**：

- [ ] Given 用户在首页, When 输入"杭州"作为目的地, Then 系统通过地图 API 联想匹配显示"杭州、杭州西湖、杭州千岛湖"等选项供选择
- [ ] Given 用户已选择目的地"杭州", When 选择天数"3"和风格标签"美食"（可多选最多 3 个，也可跳过）, Then 天数显示为 3，风格标签高亮显示"美食"
- [ ] Given 用户已完成输入, When 点击"生成行程"按钮, Then 30 秒内跳转至行程详情页，展示按天分组的行程卡片列表
- [ ] Given AI 返回结果, When 行程渲染完成, Then 每天包含 3-5 个行程节点（景点/餐厅/活动），且"美食"风格下至少包含 1 个美食相关推荐
- [ ] Given 用户未选择风格标签, When 点击"生成行程", Then AI 以"综合推荐"模式生成行程，覆盖景点、美食、休闲等多类型
- [ ] Given 用户输入无法识别的地名"xyz123", When 点击"生成行程", Then 提示"没有找到这个地方，试试输入城市名？"并展示热门城市快捷选择
- [ ] Given 用户输入天数超过 15, When 尝试选择, Then 提示"建议分段规划"，不允许选择超过 15 天

**依赖**：无（基础 Story）
**技术备注**：行程数据 Schema 需在此 Story 中定义（Trip → Day → Card → Transport），后续所有 Story 依赖此数据结构。

### US-02：生成进度反馈

**角色**：作为旅行者
**需求**：我希望在 AI 生成行程的过程中看到进度提示
**价值**：知道系统正在工作而不是卡住了，减少等待焦虑
**优先级**：P1
**Story Points**：2

**Acceptance Criteria**：

- [ ] Given 用户点击"生成行程"后, When 等待期间, Then 显示加载动画和滚动进度文案（如"正在搜索杭州的热门景点…"、"正在规划路线…"、"正在生成行程…"）
- [ ] Given 生成过程持续超过 60 秒, When 超时触发, Then 终止请求，显示"生成超时了，请再试一次"并提供"重试"按钮
- [ ] Given 用户点击"重试", When 重新发起请求, Then 重新进入生成流程，展示加载动画

**依赖**：US-01
**技术备注**：进度文案可硬编码轮播，不需要后端实时推送进度。

---

### US-03：AI 异常处理与降级方案

**角色**：作为旅行者
**需求**：我希望在 AI 服务不可用时仍能得到一份基础行程
**价值**：不因 AI 故障导致整个产品不可用
**优先级**：P1
**Story Points**：3

**Acceptance Criteria**：

- [ ] Given AI 服务不可用（503/500 错误）, When 用户点击"生成行程", Then 展示该目的地的通用行程模板，顶部 banner 提示"AI 暂时不可用，这是一份通用行程模板，你可以在此基础上调整"
- [ ] Given 网络中断, When 用户点击"生成行程", Then 自动重试 1 次；重试仍失败时显示"网络连接失败，请检查网络后重试"
- [ ] Given 通用模板已展示, When 用户编辑模板行程, Then 编辑功能（对话微调/卡片操作）正常使用（如果 AI 恢复）

**依赖**：US-01
**技术备注**：需提前准备 5-10 个热门城市的通用行程模板（JSON 格式），作为降级方案。

### US-04：历史行程复用提示

**角色**：作为旅行者
**需求**：我希望在重复规划同一目的地时，AI 能参考我之前的行程
**价值**：避免每次从零开始，在上次基础上迭代更高效
**优先级**：P2
**Story Points**：3

**Acceptance Criteria**：

- [ ] Given 用户输入目的地"杭州"且之前生成过杭州的行程, When 输入完成, Then 显示提示"你之前规划过杭州的行程（3天前），要在上次的基础上调整吗？"并提供"基于上次调整"和"重新生成"两个选项
- [ ] Given 用户选择"基于上次调整", When 确认, Then 加载上次行程数据进入行程详情页，可直接编辑
- [ ] Given 用户选择"重新生成", When 确认, Then 正常走 AI 生成流程
- [ ] Given 用户之前没有该目的地的行程, When 输入完成, Then 不显示任何提示，正常生成

**依赖**：US-01, US-19（历史行程查看）
**技术备注**：需查询本地/云端存储的历史行程数据。

## Epic 2：行程卡片系统（F2）

### US-05：按天分组卡片列表展示

**角色**：作为旅行者
**需求**：我希望行程以按天分组的卡片列表形式展示，一眼看到每天的安排
**价值**：直观浏览行程结构，快速判断每天的安排是否合理
**优先级**：P0
**Story Points**：3

**Acceptance Criteria**：

- [ ] Given 行程已生成, When 进入行程详情页, Then 默认展示时间轴视图，行程按天分组纵向排列
- [ ] Given 行程有 3 天, When 浏览列表, Then 每天有标题（Day 1 / Day 2 / Day 3），点击标题可折叠/展开该日行程
- [ ] Given 卡片列表展示中, When 查看每张卡片, Then 卡片包含：类型图标（景点/餐厅/交通/住宿）、名称、预计时间段（如"14:00-16:00"）、简介摘要（20 字内）
- [ ] Given 相邻两张卡片之间, When 查看交通信息, Then 显示交通连接标签（如"步行 15 分钟"/"打车 20 分钟"/"地铁 3 站"）
- [ ] Given 每天顶部, When 查看当日概要, Then 显示总步行距离、预计花费、景点数量

**依赖**：US-01（依赖行程数据 Schema）
**技术备注**：卡片类型图标使用 emoji 或简单 SVG，不做复杂图标系统。

---

### US-06：卡片详情面板

**角色**：作为旅行者
**需求**：我希望点击卡片后能查看景点的详细信息
**价值**：帮助判断这个景点是否值得去、是否适合我的行程
**优先级**：P0
**Story Points**：3

**Acceptance Criteria**：

- [ ] Given 行程已展示, When 点击任意卡片, Then 从底部弹出半屏详情面板（不离开当前页面）
- [ ] Given 详情面板已展开, When 查看内容, Then 展示：完整介绍、开放时间、门票价格（标注"仅供参考"）、用户评分、实景照片
- [ ] Given 某景点的 POI 详情数据加载失败, When 打开详情面板, Then 显示基础信息（名称+类型），卡片上标注"详细信息加载中…"
- [ ] Given 详情面板已展开, When 向下滑动或点击面板外区域, Then 面板关闭，回到行程列表

**依赖**：US-05
**技术备注**：照片来源于地图 POI 数据，如 POI 无照片则显示类型占位图。门票价格标注"仅供参考"是因为 AI 生成数据可能不是最新的。

---

### US-07：卡片替换、删除与标记

**角色**：作为旅行者
**需求**：我希望能在卡片详情中直接替换、删除景点或标记"必去"
**价值**：快速对单个卡片做基础操作，无需进入对话模式
**优先级**：P0
**Story Points**：2

**Acceptance Criteria**：

- [ ] Given 卡片详情面板已展开, When 点击"替换"按钮, Then AI 推荐 3 个同类型替代选项列表；用户点选后自动替换当前卡片，详情面板更新为新卡片内容
- [ ] Given 替换时无同类型推荐, When 点击"替换", Then 扩大范围推荐不同类型的热门选项，提示"没有找到类似的景点，这些热门推荐看看？"
- [ ] Given 卡片详情面板已展开, When 点击"删除"按钮, Then 卡片带滑出动画消失，下方卡片自动上移；AI 弹出提示"已删除 XX，需要我推荐替代吗？"
- [ ] Given 删除后当日行程仅剩 1 个卡片, When 删除完成, Then AI 自动提示"下午空出来了，要推荐一些活动吗？"
- [ ] Given 卡片详情面板已展开, When 点击"必去"标签, Then 卡片显示星标图标；后续 AI 调整行程时优先保留带星标的卡片

**依赖**：US-05, US-06
**技术备注**：替换推荐需调用 AI 接口，传入当前卡片类型和目的地，获取同类型推荐。

## Epic 3：对话式微调（F3）

### US-08：对话面板与核心意图识别

**角色**：作为旅行者
**需求**：我希望通过自然语言对话来调整行程，包括替换景点、新增活动、删除景点、调整顺序
**价值**：像跟朋友聊天一样轻松调整行程，不用学习复杂操作
**优先级**：P0
**Story Points**：8

**Acceptance Criteria**：

- [ ] Given 行程详情页, When 点击底部"对话调整"按钮, Then 弹出半屏对话面板，包含消息列表和文字输入框
- [ ] Given 对话面板已打开, When 用户输入"第二天下午换个户外景点"并发送, Then AI 识别为"替换"意图，定位 Day2 下午卡片，推荐 3 个户外景点选项供选择
- [ ] Given 对话面板已打开, When 用户输入"第三天加一个当地的夜市"并发送, Then AI 识别为"新增"意图，在 Day3 中插入夜市卡片，新卡片高亮显示 3 秒
- [ ] Given 对话面板已打开, When 用户输入"把博物馆去掉"并发送, Then AI 识别为"删除"意图，匹配最相关的卡片并删除，提示"已删除 XX 博物馆"
- [ ] Given 对话面板已打开, When 用户输入"把西湖放到第一天"并发送, Then AI 识别为"调整顺序"意图，移动卡片到 Day1 并重算路线
- [ ] Given AI 修改完成后, When 行程更新, Then 变更的卡片以高亮动画标记（持续 3 秒），帮助用户定位变化
- [ ] Given 用户输入为空或纯符号, When 尝试发送, Then 不发送消息，输入框抖动提示

**依赖**：US-01, US-05
**技术备注**：这是最大的单个 Story（8 Points），涉及 prompt 工程（意图识别）、上下文管理（关联行程数据）、前端对话 UI。建议先实现意图识别 prompt 并用 API 测试准确率，再做前端 UI。

---

### US-09：高级对话意图与模糊处理

**角色**：作为旅行者
**需求**：我希望 AI 能理解更复杂的调整意图（全局调整、风格调整、信息查询），并在理解不清时主动追问
**价值**：覆盖更多调整场景，对话体验更自然智能
**优先级**：P1
**Story Points**：3

**Acceptance Criteria**：

- [ ] Given 对话面板已打开, When 用户输入"整体预算再低一些", Then AI 识别为"全局调整"意图，重新评估所有卡片，将高价项替换为平替，变更卡片逐个高亮
- [ ] Given 对话面板已打开, When 用户输入"多加一些美食推荐", Then AI 识别为"风格调整"意图，在空余时段插入美食相关卡片
- [ ] Given 对话面板已打开, When 用户输入"这个景点值得去吗", Then AI 识别为"信息查询"意图，返回该景点的评价、推荐理由和注意事项
- [ ] Given 用户输入模糊意图"换个地方"（未指定哪天哪个景点）, When 发送消息, Then AI 回复确认问题"你想调整哪一天的行程？"并附带快捷选项按钮（Day1/Day2/Day3）
- [ ] Given 对话上下文已建立, When 用户引用前文"刚才推荐的那个不错，再加回来", Then AI 理解上下文，将之前推荐的景点重新加入行程

**依赖**：US-08
**技术备注**：全局调整和风格调整涉及批量修改多张卡片，需确保每次修改后都重算路线和时间。

---

### US-10：对话撤销

**角色**：作为旅行者
**需求**：我希望每次对话修改后都能一键撤销
**价值**：大胆尝试调整不怕搞砸，降低操作心理负担
**优先级**：P1
**Story Points**：2

**Acceptance Criteria**：

- [ ] Given AI 通过对话完成了一次行程修改, When 回复消息旁显示"撤销"按钮, Then 用户点击"撤销"后行程回退到修改前状态
- [ ] Given 用户点击"撤销", When 行程回退完成, Then 对话中追加一条"已撤销上次修改"的系统标记
- [ ] Given 用户连续做了 3 次修改, When 点击第一次修改旁的"撤销", Then 仅回退第一次修改，第二第三次修改保留（按时间顺序回退）
- [ ] Given 行程已是初始状态（未做任何修改）, When 查看对话面板, Then 不显示"撤销"按钮

**依赖**：US-08
**技术备注**：需维护行程修改历史栈（至少保留 10 步），每次 AI 修改 push 当前状态入栈。

## Epic 4：拖拽排序编辑（F4）

### US-11：卡片同日拖拽排序

**角色**：作为旅行者
**需求**：我希望通过手指拖拽卡片来调整同一天的游览顺序
**价值**：最直觉的排序方式，比打字说"调整顺序"更快
**优先级**：P1
**Story Points**：3

**Acceptance Criteria**：

- [ ] Given 行程详情页, When 长按某张卡片 0.5 秒, Then 卡片浮起并带轻微阴影效果，其他卡片出现蓝色插入位置指示线
- [ ] Given 卡片处于拖拽状态, When 拖动到同一天的另一位置并松手, Then 卡片插入新位置，带弹跳动画，其余卡片自动重排
- [ ] Given 拖拽完成, When 卡片落入新位置, Then 系统自动重算相邻卡片间的交通方式和时间，更新每天的预计总耗时和步行距离
- [ ] Given 拖拽完成, When 路线更新, Then 顶部弹出 toast "路线已更新" + "撤销"按钮（持续 5 秒后自动消失）
- [ ] Given 用户快速连续拖拽多张卡片, When 第一次拖拽动画未完成, Then 排队等待，第一次完成后再处理下一次

**依赖**：US-05
**技术备注**：微信小程序触摸事件（touchstart/touchmove/touchend）实现，需在真机上验证流畅度。如体验不佳，降级为"点击交换"交互。

---

### US-12：卡片跨天拖拽

**角色**：作为旅行者
**需求**：我希望把卡片拖到另一天的行程中
**价值**：跨天调整一气呵成，不用先删再加
**优先级**：P2
**Story Points**：2

**Acceptance Criteria**：

- [ ] Given 卡片处于拖拽状态, When 拖动到另一天的区域并松手, Then 卡片自动归入该天行程，该天和原行程天的交通和时间自动重算
- [ ] Given 跨天拖拽导致目标天行程时间超过 14 小时（8:00-22:00）, When 卡片落入, Then 目标天顶部标红"行程过满"，但不阻止操作
- [ ] Given 卡片被拖到深夜时段, When 落入, Then 卡片标注提醒"注意：这个景点通常白天开放"
- [ ] Given 跨天拖拽完成, When 路线更新, Then 顶部弹出 toast + "撤销"按钮

**依赖**：US-11
**技术备注**：跨天拖拽需要滚动到目标天区域，需处理自动滚动逻辑（拖拽靠近屏幕边缘时自动滚动列表）。

## Epic 5：地图视图（F5）

### US-13：地图路线展示与日期聚焦

**角色**：作为旅行者
**需求**：我希望在地图上看到行程路线，按天查看不同颜色的路线连线
**价值**：直观判断景点间距离和路线合理性，发现"绕远路"的问题
**优先级**：P1
**Story Points**：5

**Acceptance Criteria**：

- [ ] Given 行程详情页, When 点击顶部"地图"Tab, Then 页面切换为全屏地图视图，所有行程点以带序号气泡标注
- [ ] Given 地图视图展示中, When 查看气泡, Then 按天着色（Day1 蓝色、Day2 绿色、Day3 橙色…），同一天内景点间用虚线连接并标注交通方式和预估时间
- [ ] Given 地图展示多天行程, When 地图初始化, Then 默认缩放级别为"能看到所有行程点"
- [ ] Given 地图视图展示中, When 点击顶部日期 Tab（Day1/Day2/Day3）, Then 地图聚焦该天路线，缩放到合适级别，其他天气泡灰显
- [ ] Given 地图视图展示中, When 点击"全部"Tab, Then 显示所有天的路线（不同颜色）

### 🆕 US-20：POI 数据实时富化

**角色**：作为旅行者
**需求**：我希望 AI 推荐的每个景点都有真实的评分、照片和价格，而不是凭空捏造的信息
**价值**：解决"AI 会不会瞎编景点"的核心信任问题——这是用户是否持续使用的基础
**优先级**：P1
**Story Points**：5

**Acceptance Criteria**：

- [ ] Given AI 生成行程完成, When 云函数处理完成, Then 至少 70% 的卡片被成功富化（_poiStatus: "verified"）
- [ ] Given 高德 API 正常, When 行程卡片被富化, Then 卡片包含：真实评分（标注"高德评分"）、人均消费、实景照片 URL、精确坐标、商圈、特色菜标签
- [ ] Given 某个 POI 在高德中搜索无结果, When 富化失败, Then 该卡片标记 _poiStatus: "not_found"，保留 AI 原始数据，不阻塞行程返回
- [ ] Given 高德 API 超时（>5s）, When 查询超时, Then 该卡片跳过富化，标记 _poiStatus: "api_error"，继续处理下一个卡片
- [ ] Given 高德 API 全挂（500/503）, When 所有查询失败, Then 全部卡片标记 _dataSource: "ai"，正常返回行程，不报错
- [ ] Given 同一景点的卡片被多次查询, When 第 2+ 次查询, Then 命中云数据库缓存（Key=高德 POI ID），响应 <100ms

**依赖**：US-01
**技术备注**：富化层在 ai-chat 云函数内执行；使用 p-limit(5) 并发查询高德；城市约束强制开启；缓存 7 天 TTL。Prompt 需增加名称标准化约束以提升匹配率。

### 🆕 US-21：富化卡片渐进渲染

**角色**：作为旅行者
**需求**：我希望在等待 POI 数据加载时能看到卡片基本内容，而不是一直盯着加载动画
**价值**：渐进式展示降低感知等待时间——卡片从"骨架"状态逐步"点亮"
**优先级**：P1
**Story Points**：2

**Acceptance Criteria**：

- [ ] Given 行程生成完成, When POI 富化仍在进行中, Then 页面先展示卡片骨架态（灰色占位、文字已有、照片位显示类型图标）
- [ ] Given POI 数据就绪, When 某张卡片富化完成, Then 该卡片从骨架态"点亮"——照片浮现（带 fade-in 动画）、评分标签弹出、商圈/标签文字出现
- [ ] Given 某张卡片富化失败（_poiStatus ≠ "verified"）, When 渲染, Then 显示统一"信息收集中"占位态（类型图标 + 灰色背景），不暴露空白字段
- [ ] Given 照片 URL 加载失败, When `<image>` binderror 触发, Then 显示类型占位图（景点/餐厅/购物各一套默认图）

**依赖**：US-05, US-20
**技术备注**：前端需要区分 loading/success/error 三个卡片渲染态；使用 CSS transition/fade-in 动画；照片占位图作为小程序静态资源打包。
