# 个人旅途规划应用 — 简单 PRD

## 1. 项目信息

- **Language**: 中文
- **Programming Language**: Vite + React + MUI + Tailwind CSS
- **Project Name**: `trip_planner`
- **原始需求复述**: 用户希望构建一个个人旅途规划工具，核心能力包括：AI 智能生成行程建议、手动调整行程、输入目的地后获取当地美食与景点信息。面向个人使用，强调实用性与个性化。

---

## 2. 产品定义

### Product Goals

1. **智能行程生成**：利用 AI 根据用户输入的目的地、时间、偏好，一键生成合理行程规划，降低行程规划的认知负荷。
2. **灵活手动调整**：提供直观的拖拽/编辑交互，让用户在 AI 生成的行程基础上自由增删改，确保行程完全符合个人意愿。
3. **目的地信息聚合**：输入目的地后自动聚合"好吃的"（美食/餐厅）和"好玩的"（景点/活动）信息，为行程填充提供高质量素材。

### User Stories

1. As a **旅行爱好者**, I want to **输入目的地和旅行天数，AI 自动生成一份行程草案** so that 我不用花大量时间查资料做初步规划。
2. As a **旅行爱好者**, I want to **在 AI 生成的行程上拖拽调整日程顺序、增删景点** so that 行程能完全按我的偏好和实际情况灵活安排。
3. As a **旅行爱好者**, I want to **查看目的地推荐美食和景点列表** so that 我能快速发现值得体验的内容并添加到行程中。
4. As a **旅行爱好者**, I want to **为行程中的每个项目添加备注（如预订号、开放时间等）** so that 旅行中我能快速查看关键信息。
5. As a **旅行爱好者**, I want to **保存和管理多个旅行计划** so that 我可以复用或对比不同行程方案。

**布局说明**：

- **左侧面板**（约 30% 宽度）：目的地推荐区，分"好吃的"和"好玩的"两个 Tab，每个条目可点击"添加到行程"一键加入右侧某天
- **右侧面板**（约 70% 宽度）：行程时间轴，按天分段展示；支持拖拽排序项目、点击编辑内容/备注、删除项目
- **底部 AI 对话区**：可向 AI 提问或指令（如"帮我把 Day 2 改成更休闲的节奏"），AI 返回修改建议后用户确认应用

        │

### Open Questions

1. **AI 集成方式**：作为个人工具，建议优先采用 **大模型 API 直接调用**（如 OpenAI / DeepSeek / Claude API），用户自行配置 API Key。好处是轻量、无需后端、生成质量高；后续可扩展为本地模型（Ollama）以降低成本。是否需要支持多种 API 供应商切换？

2. **目的地数据获取**：美食和景点推荐有三种来源可组合：
   - **AI 生成**：最简单，让大模型根据目的地直接输出推荐列表（质量依赖模型，但零成本集成）
   - **地图 API**（如腾讯地图/高德）：可获得评分、人均、营业时间等结构化数据，但需申请 API Key
   - **旅游平台 API**：数据更丰富但接入复杂
   - **建议**：P0 阶段优先使用 AI 生成推荐 + 基础结构化数据，P1 阶段接入地图 API 补充评分/位置等字段。用户倾向哪种？

3. **手动调整与 AI 协同 UX**：当用户手动修改行程后，AI 再次优化时如何处理冲突？建议策略：
   - AI 优化仅作用于用户选定的范围（某天/某段）
   - 用户手动修改的项目标记 `source: "manual"`，AI 优化时默认保留手动项目不覆盖
   - 提供版本快照，用户可回退到任意历史状态

4. **是否需要后端服务**：纯前端（localStorage）能满足个人使用场景吗？如果未来要加分享/协作功能，则需要后端。当前建议纯前端先行。

5. **移动端适配**：作为个人工具，用户是否需要手机端使用？如果是，MUI 的响应式布局可覆盖，但拖拽交互在移动端体验需要额外设计。

6. **行程时间粒度**：AI 生成的行程是否需要精确到小时（如 09:00-11:00），还是只按上午/下午/晚上分段？精确到小时更实用但 AI 生成准确度有限。
