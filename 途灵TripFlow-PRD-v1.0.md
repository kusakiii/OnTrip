## 1. 背景与目标

### 1.1 背景

休闲度假旅行者在规划行程时面临两个核心困境。第一是信息过载：攻略平台（小红书、马蜂窝、携程）信息量巨大且质量参差不齐，用户平均花费 3-5 小时才能拼凑出一份行程，且不确定推荐是否可靠。第二是行程僵化：一旦排好行程，旅途中遇到天气变化、体力不支或临时发现好去处时，缺乏快速调整的工具支持，只能靠人肉重新搜索和决策。

当前市场上的 AI 旅行工具（Wanderlog、携程问道、飞猪问一问、ChatGPT）在"生成行程"上已有不错的能力，但普遍存在三个短板：生成的行程难以灵活编辑（只能重新生成）、缺乏空间感知（不计算景点间距离和时间）、旅途中完全失能（行程生成后 AI 就消失了）。

现在是做这个产品的时机：大模型 API 能力已成熟（GPT-4、Claude 等均支持结构化输出），微信小程序开发框架稳定，POI 数据源（高德/腾讯地图）开放度高。作为个人工具，可以用最低成本验证"AI 生成 + 人工微调"这个核心假设。

> ⚠️ **数据来源说明**：背景中引用的定量论断（如"3-5 小时规划"等）基于个人经验和行业观察，尚未经过大规模用户调研验证。建议在 Phase 1 开发前进行 5-8 个目标用户的轻量访谈，验证"信息过载"和"行程僵化"这两个核心假设。

### 1.2 目标用户

| 用户角色               | 特征描述                                              | 核心诉求                                     | 使用场景                                    |
| ---------------------- | ----------------------------------------------------- | -------------------------------------------- | ------------------------------------------- |
| 休闲度假旅行者（主要） | 每年出游 2-4 次，偏好自由行，不愿意花大量时间做攻略   | 快速获得一份靠谱的行程，并能在旅途中灵活调整 | 出行前 1-2 周规划行程；旅途中每日查看和调整 |
| 周末短途旅行者（次要） | 时间紧张，通常只有 2-3 天，需要在有限时间内最大化体验 | 高效规划，减少决策时间                       | 周四/周五快速规划周末行程                   |

## 2. 需求概述

AI 根据极简输入（目的地+天数+风格）一键生成可编辑的行程卡片，用户通过对话和手动两种方式灵活调整，并在旅途中随时查看行程。

> **V1.0 调整说明（2026-07-02）**：F4 拖拽排序已从 V1.0 移除。理由：(1) 小程序中实现流畅拖拽的技术风险和工时不匹配个人项目定位；(2) F3 对话微调已覆盖"调整顺序"场景；(3) 节省的工时用于 DTO 校验层和用户体验增强。

## 3. 功能详细设计

### 3.1 F1：极简输入 & 一键生成行程

**功能描述**：用户输入目的地、天数和旅行风格标签，AI 在 30 秒内生成一份完整的结构化行程。
**用户故事**：作为一个休闲旅行者，我希望只花 10 秒钟告诉 AI 我想去哪里、去几天、喜欢什么风格，就能得到一份完整的行程草案，省去翻攻略的时间。
**优先级**：P0

**业务规则**：

1. 目的地支持输入城市名或景区名（如"杭州"、"西湖"），系统通过地图 API 自动匹配并确认，匹配不到时提示用户补充
2. 天数范围为 1-15 天，超出 15 天时提示"建议分段规划"
3. 旅行风格标签提供预设选项：文艺、美食、亲子、自然、购物、休闲、暴走，支持多选（最多 3 个），可跳过不选
4. AI 生成的行程包含：每日行程概要、每个时段的景点/活动/餐厅推荐（含名称、简介、预计停留时间、开放时间）、景点间交通方式和预估时间
5. 生成结果以结构化 JSON 返回，**后端云函数须对返回结果做 JSON Schema 校验**（缺失字段赋默认值、非法值拒绝或修复、格式异常自动重试 1 次），校验通过后前端渲染为行程卡片列表
6. 生成过程中展示加载动画和进度文案（"正在搜索杭州的热门景点…"、"正在规划路线…"），最长等待 60 秒
7. 如用户之前生成过同一目的地的行程，提示"你之前规划过杭州的行程，要在上次的基础上调整吗？"

**交互流程**：

1. 用户从小程序首页进入，看到极简输入面板：目的地输入框 + 天数选择器 + 风格标签栏 + "生成行程"按钮
2. 用户输入目的地，系统自动联想匹配（如输入"杭"显示"杭州、杭州西湖、杭州千岛湖"）
3. 用户选择天数（滚轮选择器，1-15）
4. 用户点击风格标签（可选，不选则 AI 默认"综合推荐"）
5. 点击"生成行程"按钮，进入生成等待页（加载动画 + 进度文案）
6. 成功：跳转至行程详情页，展示完整的行程卡片列表
7. 失败（超时或异常）：展示"生成超时，请重试"提示，提供"重试"按钮

**异常处理**：

| 异常场景             | 处理方式                                         | 用户提示                                                    |
| -------------------- | ------------------------------------------------ | ----------------------------------------------------------- |
| 目的地无法识别       | 引导用户输入更具体的地名，或从热门城市列表中选择 | "没有找到这个地方，试试输入城市名？"                        |
| AI 生成超时（>60秒） | 终止请求，允许重试                               | "生成超时了，请再试一次"                                    |
| AI 服务不可用        | 展示降级方案：提供该目的地的通用行程模板         | "AI 暂时不可用，这是一份通用行程模板，你可以在此基础上调整" |
| 网络中断             | 自动重试 1 次，失败后提示                        | "网络连接失败，请检查网络后重试"                            |

### 3.2 F2：行程卡片系统

**功能描述**：将 AI 生成的行程以结构化卡片形式呈现，每张卡片代表一个行程节点，支持浏览详情和基础操作。
**用户故事**：作为一个旅行者，我希望行程以清晰的卡片形式展示，一眼就能看到每天的安排，点开卡片能查看每个景点的详细信息。
**优先级**：P0

**业务规则**：

1. 行程按"天"分组，每天包含若干行程卡片，按时间顺序排列
2. 每张卡片包含：类型图标（景点/餐厅/交通/住宿）、名称、预计时间段（如"14:00-16:00"）、简介摘要（20字内）
3. 卡片间显示交通连接信息（步行 X 分钟 / 打车 X 分钟 / 地铁 X 站）
4. 每天顶部显示当日概要：总步行距离、预计花费、景点数量
5. 点击卡片展开详情面板：完整介绍、开放时间、门票价格（标注"仅供参考"）、用户评分、实景照片（来源于地图 POI 数据）
6. 每张卡片右上角提供操作菜单：替换（AI 推荐同类型替代）、删除、标记为"必去"/"可去可不去"

**交互流程**：

1. 行程生成后，用户进入行程详情页，默认展示"时间轴视图"
2. 页面按天纵向排列卡片，支持上下滑动浏览
3. 点击某天标题可折叠/展开该日行程
4. 点击单张卡片，从底部弹出详情半屏面板（不离开当前页面）
5. 详情面板中点击"替换"，AI 推荐 3 个同类型替代选项，用户点选后自动替换当前卡片
6. 详情面板中点击"删除"，卡片带滑出动画消失，下方卡片自动上移，AI 提示"已删除 XX，需要我推荐替代吗？"
7. 详情面板中点击"必去"标签，卡片显示星标，后续调整时 AI 会优先保留带星标的卡片

**异常处理**：

| 异常场景                 | 处理方式                                      | 用户提示                                 |
| ------------------------ | --------------------------------------------- | ---------------------------------------- |
| 替换时无同类型推荐       | 扩大搜索范围，推荐不同类型的热门选项          | "没有找到类似的景点，这些热门推荐看看？" |
| 删除导致当日行程过于空旷 | AI 自动提示补充建议                           | "下午空出来了，要推荐一些活动吗？"       |
| 卡片详情数据加载失败     | 显示基础信息（名称+类型），标注"详情暂不可用" | 卡片上显示"详细信息加载中…"              |

### 3.3 F3：对话式微调

**功能描述**：用户通过自然语言对话，对已有行程进行调整，AI 理解意图后修改对应行程卡片。
**用户故事**：作为一个旅行者，我希望像跟朋友聊天一样说"第二天下午换个轻松点的"或"预算再少一点"，AI 就能帮我调整行程。
**优先级**：P0

**业务规则**：

1. 对话入口常驻在行程详情页底部，点击展开对话面板（半屏）
2. 支持以下对话意图类型：
   - **替换**："第二天下午换一个户外景点" → AI 识别目标为 Day2 下午卡片，推荐替代方案
   - **新增**："第三天加一个当地的夜市" → AI 在 Day3 中插入新卡片
   - **删除**："把博物馆去掉" → AI 匹配最相关的卡片并删除
   - **调整顺序**："把西湖放到第一天" → AI 调整对应卡片位置并重算路线
   - **全局调整**："整体预算再低一些" → AI 重新评估所有卡片，替换高价项
   - **风格调整**："多加一些美食推荐" → AI 在空余时段插入美食相关卡片
   - **信息查询**："这个景点值得去吗" → AI 返回该景点的评价和推荐理由
3. AI 每次修改后，用高亮动画标记变更的卡片（持续 3 秒），帮助用户定位变化
4. 对话上下文保持整个行程会话期间，支持引用前文（"刚才推荐的那个不错，再加回来"）
5. AI 每次回复附带"撤销"按钮，用户可一键回退到上一步操作

**交互流程**：

1. 用户点击行程底部的"对话调整"按钮，弹出半屏对话面板
2. 输入框支持文字输入（暂不支持语音）
3. 用户发送消息后，AI 在 3-8 秒内返回回复
4. 如果 AI 理解意图明确，直接执行修改并展示变更结果（卡片高亮标记变化部分）
5. 如果 AI 理解不确定（如"换个地方"未指定哪天），回复确认问题（"你想调整哪一天的行程？"），附带快捷选项按钮
6. 用户点击"撤销"按钮，行程回退到上一步状态，对话中追加"已撤销"标记

**异常处理**：

| 异常场景                             | 处理方式                         | 用户提示                                                   |
| ------------------------------------ | -------------------------------- | ---------------------------------------------------------- |
| AI 无法理解意图                      | 给出理解建议，引导用户更具体描述 | "你是想调整第二天的行程吗？可以说得更具体一些"             |
| 修改导致路线不合理（如景点间距过远） | AI 标注风险提醒，但不阻止修改    | "这两个景点相距 15 公里，打车约 30 分钟，确认这样安排吗？" |
| 对话服务超时                         | 自动重试 1 次                    | "回复稍慢，正在努力思考中…"                                |
| 用户输入为空或纯符号                 | 不发送，输入框抖动提示           | 无                                                         |

5. 如果拖拽导致当日行程时间溢出（超出合理活动时间 8:00-22:00），顶部标红提示"当日行程过满"

1. 用户在行程详情页，长按某张卡片
2. 卡片浮起，其他卡片出现插入位置指示线（蓝色横线）
3. 用户拖动卡片到目标位置
4. 松手，卡片落入新位置，带弹跳动画
5. 系统自动重算路线，更新交通信息标签
6. 顶部出现 toast "路线已更新" + "撤销"按钮（持续 5 秒后自动消失）

### 3.5 F5：地图视图（V1.0 缩减版）

**功能描述**：在地图上标注行程点的位置，帮助用户直观判断景点间的空间关系。**V1.0 仅做静态标注**（气泡 + 距离标注），不做路线连线、拖拽联动和日期聚焦动画。
**用户故事**：作为一个旅行者，我希望在地图上看到景点的位置分布，一眼就知道景点之间远不远。
**优先级**：P1（缩减版）

**业务规则**：

1. 行程详情页顶部提供"时间轴 / 地图"双视图切换 Tab
2. 地图视图显示：所有行程卡片以带序号的气泡标注在地图上（按天着色，Day1 蓝色、Day2 绿色…）
3. 气泡间显示直线距离标注（如"1.2km"），不做路线连线
4. 点击地图气泡，弹出对应行程卡片的迷你信息卡片（名称+时间+类型）
5. 地图默认缩放级别为"能看到所有行程点"，支持手动缩放和平移

> **V2.0 扩展方向**：路线连线、日期聚焦动画、地图拖拽联动

**交互流程**：

1. 用户在行程详情页点击"地图"Tab
2. 页面切换为全屏地图，行程路线叠加显示
3. 顶部保留日期 Tab（Day1/Day2/Day3…），点击某天地图聚焦该天路线
4. 点击"全部"Tab 显示所有天的路线（不同颜色）
5. 点击某个地图气泡，底部弹出对应卡片迷你信息
6. 在地图视图中长按气泡可进入拖拽模式

**异常处理**：

| 异常场景                     | 处理方式                           | 用户提示                     |
| ---------------------------- | ---------------------------------- | ---------------------------- |
| 地图加载失败                 | 显示静态地图占位图 + 重试按钮      | "地图加载失败，点击重试"     |
| 景点坐标缺失（POI 数据不全） | 在地图上不标注该点，路线跳过该节点 | 卡片上标注"位置信息暂不可用" |
| 行程点过于密集导致气泡重叠   | 自动聚合为簇（Cluster），点击展开  | 无                           |

### 3.6 F6：行程管理（保存/导出/历史）

**功能描述**：用户可以保存行程、导出分享、查看历史行程记录。
**用户故事**：作为一个旅行者，我希望保存规划好的行程，旅途中随时打开看，也能方便地分享给同伴。
**优先级**：P1

**业务规则**：

1. **保存**：行程自动保存（每次编辑操作后 2 秒自动触发），同时支持手动点击"保存"按钮
2. **导出**：支持两种导出格式：
   - 图片长图：将行程渲染为精美长图，适配微信朋友圈分享尺寸
   - 小程序分享卡片：发送给微信好友/群，对方打开可查看行程（只读）
3. **历史行程**：首页"我的行程"列表，按创建时间倒序排列，每条显示：目的地、天数、创建日期、缩略图

**交互流程**：

1. 行程详情页右上角"…"菜单提供：保存、导出长图、分享给好友、复制行程、删除行程

### 3.7 F7：行程 POI 数据实时富化（🆕 V1.0 新增）

**功能描述**：AI 生成行程后，系统自动调用高德地图 POI API 校验每个推荐的真实存在性，并补全评分、人均消费、实景照片、精确坐标、营业时间、商圈、特色菜标签等字段。解决"AI 会不会瞎编景点"的核心信任问题。
**用户故事**：作为一个旅行者，我希望 AI 推荐的每个景点都有真实的评分、照片和价格，而不是凭空捏造的信息。
**优先级**：P1

**业务规则**：

1. 富化在 AI 生成完成后、DTO 校验通过后自动触发，不需要用户手动操作
2. 调用高德地图 POI 搜索 API（`extensions=all`），按城市强制约束搜索
3. 名称匹配策略：精确匹配 → 包含匹配 → 编辑距离 < min(3, 名称长度×0.3) → 降级
4. 富化失败**绝不阻塞行程返回**：单卡片失败静默降级，API 全挂时全部标记 AI 数据源
5. 富化成功的卡片展示：真实评分（标注"高德评分"）、人均消费、实景照片、商圈、特色菜标签
6. 富化失败的卡片展示：AI 原始信息 + 统一"信息收集中"占位态，不暴露空白字段
7. 照片直接使用高德 URL（不转存），加载失败显示类型占位图
8. 替换推荐时 AI 返回候选列表（3 个选项），后端对候选列表做批量富化后展示

**技术实现要点**：

- 并发查询（上限 5 并发），避免串行超时
- 缓存 Key 使用高德 POI ID（非 AI 名称），7 天 TTL
- Provider 接口抽象，方便未来换腾讯地图或其他数据源
- API Key 存云函数环境变量，前端不接触

**异常处理**：

| 异常场景                 | 处理方式                                     | 用户提示                         |
| ------------------------ | -------------------------------------------- | -------------------------------- |
| 高德 API 超时（>5s）     | 跳过该卡片，继续处理下一个                   | 卡片显示"信息收集中"占位态       |
| 高德 API 全挂（500/503） | 全部卡片标记 AI 数据源，正常返回             | 无感知（卡片以 AI 原始数据展示） |
| 单个 POI 搜索无结果      | 标记 `_poiStatus: "not_found"`，保留 AI 数据 | 卡片显示"信息收集中"占位态       |
| 照片 URL 加载失败        | 前端 `<image>` binderror 展示类型占位图      | 照片位显示景点/餐厅分类默认图    |

## 9. V2.0 待验证假设列表

> **修订说明**：V2.0 规划在此保留为方向参考，但最终优先级取决于 V1.0 的实际使用数据。每项标注"需要 V1.0 的什么数据来验证"。

V1.0 上线并经过 2-3 次真实旅行验证后，根据使用反馈和数据分析选择以下方向迭代：

### 9.1 旅途实时助手（V2.0 核心）

**方向描述**：在旅途执行阶段，AI 助手持续运行，从"规划工具"升级为"随行服务"。

**计划功能**：

| 功能         | 描述                                                                             | 优先级 |
| ------------ | -------------------------------------------------------------------------------- | ------ |
| 天气触发备选 | 接入天气 API，检测到降雨/极端天气时，自动将户外行程替换为室内备选并推送通知      | P0     |
| 路线实时优化 | 基于用户当前 GPS 位置和剩余行程，建议更优的游览顺序                              | P1     |
| 即时问答     | "附近有什么好吃的"、"这个景点排队多久"等实时问题，AI 结合搜索引擎和 POI 数据回答 | P0     |
| 应急信息     | 提供当地紧急联系电话、最近医院/警察局、领事馆信息                                | P2     |

### 9.2 弹性行程块

**方向描述**：打破精确到小时的排程，改为"上午块/下午块/晚间块"的弹性时间块，每块提供 2-3 个可选项，让用户按心情选择而非被固定时间绑死。

### 9.3 一键风格切换 & 行程对比

**方向描述**：对同一份行程，一键切换"文艺版/美食版/暴走版/佛系版"，AI 自动重新编排。或同时生成 2-3 套不同风格方案，并排对比选择。

### 9.4 反幻觉可信推荐

**方向描述**：每条推荐附带推荐理由、信息来源、可信度评分。接入小红书/大众点评等 UGC 数据做交叉验证，标注"本地人常去"标签，提升推荐可靠性。

### 9.5 旅行人格画像

**方向描述**：基于历史行程和偏好数据，构建个人旅行画像（偏好风格、预算倾向、体力水平、饮食口味），越用推荐越精准。

### V2.0 建议优先级

旅途实时助手 > 反幻觉可信推荐 > 弹性行程块 > 一键风格切换 > 旅行人格画像

判断依据：旅途实时助手直接解决"行程中无法灵活调整"这一核心痛点，是 V1.0 的自然延伸；反幻觉可信推荐解决信任问题，是用户持续使用的关键。
