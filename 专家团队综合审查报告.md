## 二、各专家关键发现

### 2.1 产品经理（Alex）— TOP 3 发现

1. **F4 优先级矛盾**：PRD 说 P0 但 ICE 说最不值得做 → 必须统一
2. **缺乏一手用户证据**："3-5 小时规划"、"91% 用 AI"等数据无来源标注
3. **成功指标缺结果性指标**：只有过程指标（满意度、耗时），没有 30 天回访率、行程采纳率等北极星指标

### 2.2 微信小程序开发者 — TOP 3 发现

1. **React → 原生迁移成本被低估**：Tailwind 无原生支持，Lucide Icons 需替换，`useEffect` 需改为生命周期
2. **地图组件限制**：`map` 组件自定义能力有限（标注样式、连线粗细、气泡聚合），Wanderlog 级体验不可能达到
3. **审核风险**：AI 生成内容类小程序需"信息资讯"类目资质，个人主体可能无法通过

### 2.3 软件架构师 — TOP 3 发现

1. **Schema 语义歧义**：`transportToNext`（当前→下一个）vs 原型渲染的 `transport`（上一个→当前）方向相反
2. **多模型 fallback 未做 prompt 兼容性验证**：DeepSeek 和 Qwen 对同一 prompt 的输出格式可能不同
3. **建议 3 个 ADR**：(a) 前端框架选择 (b) 离线数据同步策略 (c) AI 多模型路由策略

### 2.4 后端架构师 — TOP 3 发现

1. **云函数安全**：API Key 在云函数环境变量中相对安全，但需增加调用频率限制防滥用
2. **上下文管理缺失设计**：多轮对话的行程数据如何高效传入 prompt（全量 vs 摘要），token 消耗预估未做
3. **缺少数据生命周期管理**：用户数据删除、账号注销流程未定义

### 2.5 UX 架构师 — TOP 3 发现

1. **双面板互斥问题**：卡片详情和聊天面板可同时打开，z-index 冲突
2. **触控目标偏小**：部分按钮（如折叠箭头、关闭按钮）不到 44×44pt 的最小触控标准
3. **缺少空状态设计**：首次用户无历史行程时的首页状态未设计

### 2.6 Prompt 工程师 — TOP 3 发现

1. **结构化输出需要约束前缀**：prompt 中预填 `{"itinerary": [` 可显著提升 JSON 稳定性
2. **意图识别建议双阶段**：先分类意图（替换/新增/删除…），再执行具体操作，大幅提升准确率
3. **行程数据传递用"骨架摘要"**：不要全量 JSON 传入 prompt（token 爆炸），用压缩格式（"Day1(09-11 西湖→11-13 知味观)"）

### 2.7 代码审查专家 — TOP 3 发现

1. **🔴 Critical**：`styleTags` 标识符不一致 → 行程页 Header 风格标签静默不显示
2. **🟠 High**：`handleChatSend` 中 `setTimeout` 无清理 → 组件卸载后仍 setState
3. **🟠 High**：15 个 `useState` + 深层不可变更新 → 建议用 `useReducer` 统一管理

## 四、关键决策记录

### 决策 1：F4 拖拽排序 — 从 V1.0 移除

**理由**：

- ICE 评分最低（10 分），PRD 误标 P0
- 小程序 `movable-view` 拖拽体验差，5 人天投入有返工风险
- F3 对话微调已覆盖"调整顺序"（"把西湖放到第一天"）
- 5 人天可做更有价值的事（静态地图 1.5 天 + onboarding 1 天 + prompt 测试 2.5 天）

**影响**：US-11、US-12 延迟到 V2.0

### 决策 2：F5 地图 — 降级为静态标注版

**理由**：

- 竞品 Wanderlog 的地图体验在小程序原生组件中无法复现
- ICE 中 Ease=7 的前提是"轻量实现"，做全套地图交互太贵
- 用户可以在高德地图自行查看路线（PRD 自己也承认"不是不可替代"）

**方案**：标注点位（气泡）+ 直线距离标注，不做连线、不做拖拽、不做日期聚焦动画

### 决策 3：前端技术栈确认为原生微信小程序

**理由**：

- 个人工具无需跨端，单端发布足够
- 地图和手势交互对性能敏感，原生最优
- React 原型仅作交互参考，生产代码用 WXML/WXSS 重写
- 确认前需做一个首页输入面板的原生小程序还原 PoC
